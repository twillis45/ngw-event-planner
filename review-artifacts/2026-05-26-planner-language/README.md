# Sprint 21 — Planner Language Authenticity + Relationship Cognition

The planner equivalent of page 50 (Runtime Language Authenticity). Runtime OS scrubs internal doctrine from operator-facing copy. Planner OS scrubs runtime/escalation DNA from orchestration-facing language.

## What this is

This is NOT a visual redesign. This is NOT analytics expansion. This is planner-native cognition refinement — making the orchestration layer speak the language planners actually use.

## Critical insight

Planners do NOT primarily think: "what is failing?"

They think:
1. What still needs alignment?
2. What could create friction?
3. What needs confirmation?
4. What might affect flow?
5. Where is confidence weakening?
6. What needs smoothing before event day?

That distinction changes the entire cognitive system.

## Runtime-language drift identified and corrected

| Runtime DNA (banned) | Planner Native (approved) | Why |
|---|---|---|
| Unresolved | Awaiting confirmation | Planners track what needs confirming, not what is "broken" |
| Escalation | Coordination needed | Planners coordinate; they do not escalate until day-of |
| Active issue | Outstanding item | Items are outstanding, not "active issues" |
| Suppression | Deferred visibility | Planner OS breathes; it defers, it does not suppress |
| Operational delay | Timing pressure | Delays are judgments; timing pressure is observable fact |
| Containment | Alignment needed | Planners align parties; containment is crisis vocabulary |
| Incident | Coordination gap | Gaps need closing; incidents need investigating |
| Threat | Potential friction | Friction can be smoothed; threats must be neutralized |
| Recovery | Sequence adjustment | Planners adjust sequences; recovery implies damage |
| Emergency | Immediate attention | Attention is directed; emergencies trigger panic |
| Status: Critical | Needs resolution today | Actionable timeframe, not severity theater |
| Status: Warning | Approaching deadline | Temporal awareness, not threat assessment |
| Severity tier | Priority level | Priority is orchestration; severity is incident language |
| Command authority | Coordination lead | Leading coordination, not commanding operations |
| Dispatch | Assign / direct | Planners assign roles; dispatchers send units |
| Operator | Planner / coordinator | The person orchestrating is a planner, not an operator |

## New approved planner vocabulary

- **Outstanding items** — things needing attention (not "unresolved issues")
- **Awaiting confirmation** — vendor or client has not confirmed
- **Coordination needed** — parties need alignment
- **Approaching deadline** — temporal awareness without panic
- **Client decision needed** — specific approval gate
- **Timing pressure** — observable schedule constraint
- **Potential friction** — something that could affect flow
- **Fragile window** — timing with zero or minimal margin
- **Turnover pressure** — room flip or vendor swap under time constraint
- **Adjacency coordination** — vendors sharing space or timing
- **Sequence adjustment** — timeline needs re-ordering
- **Readiness** — overall preparation state

## Relationship cognition framework

Six domains of operational relationship memory:

1. **Client trust memory** — communication preference, decision pace, stress patterns, approval patterns, reassurance frequency
2. **Vendor responsiveness** — response window, contact preference, confirmation reliability, punctuality, escalation threshold
3. **Hospitality quality** — setup consistency, flexibility, guest-facing standards, problem-solving autonomy, vendor collaboration
4. **Expectation management** — vision alignment, budget calibration, scope creep, stakeholder dynamics, presentation timing
5. **Emotional continuity** — stress patterns, trust-building moments, friction resolution, celebration style, follow-up preferences
6. **Coordination friction** — vendor pairing compatibility, venue quirks, timing dependencies, communication chain reliability

## Client-confidence doctrine

Five pillars: approvals, expectations, pacing, reassurance, visibility.

Language rules:
- AVOID: "touch base", "checking in", "per our conversation", "action items", "we're on track", "circle back", "ping me"
- USE: specific confirmations, clear asks with deadlines, concrete progress, proactive contingency awareness, specific commitments

## Timeline intelligence doctrine

Not calendar software. Not task management. Living orchestration sequencing.

Six domains: dependency-aware sequencing, fragile timing windows, setup adjacency, guest-flow choreography, transition choreography, venue bottlenecks.

Mental model: live show production, not project management.

## Drift risks

1. **CRM drift** — Client confidence could become pipeline tracking. The surface shows trust memory, not deal stages.
2. **Calendar drift** — Timeline intelligence could become a scheduling app. It shows dependency-aware sequencing, not time slots.
3. **Analytics drift** — Portfolio pressure could become metrics dashboards. It shows orchestration topology, not charts.
4. **Vendor management drift** — Relationship memory could become a vendor CRM. It shows orchestration truths, not contact management.
5. **Project management drift** — Outstanding items could become task boards. They show coordination awareness, not to-do lists.
6. **Corporate communication drift** — Client language could become customer-service scripts. It shows hospitality-grade precision, not templates.

## What is NOT validated

- No real planner has reviewed these surfaces or vocabulary. All content is synthetic.
- The vocabulary matrix is a hypothesis — actual planner language patterns may differ.
- Relationship cognition domains have not been validated against real planner workflows.
- Client-confidence pillars are theoretical until field sessions confirm.
- Timeline intelligence fragile-window model has not been tested with real event sequences.
- None of these surfaces exist in code yet. They are Figma explorations.

## Honest read

This sprint corrects the single most important remaining flaw in Planner OS: it still partially spoke runtime language. "Unresolved" became "awaiting confirmation." "Escalation" became "coordination needed." "Containment" became "alignment needed." The vocabulary shift is small but the cognitive shift is fundamental — the system now thinks like an orchestrator, not a dispatcher. The five new surface types (client confidence, timeline intelligence, vendor relationship memory, portfolio pressure, event orchestration refinement) expand the planner cognitive system from a single orchestration view into a complete set of planner-native tools. Whether elite planners would actually recognize themselves in these surfaces is unknown.
