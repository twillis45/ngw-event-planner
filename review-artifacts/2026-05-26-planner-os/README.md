# Planner OS — Doctrine + Exploratory Surfaces

The orchestration layer above runtime containment. Planner OS optimizes for **confidence**. Runtime OS optimizes for **control**.

## What this is

This is not a redesign sprint. This is not "make prettier dashboards." This is the formal creation of the planner cognitive system for the product.

## Four-Layer Cognitive Architecture

| Layer | Optimizes for | Emotional mode | Time horizon |
|---|---|---|---|
| **Planner OS** | Confidence | Calm, elite, prepared | Days/weeks ahead |
| **Runtime OS** | Control | Focused, decisive, urgent | Now (seconds/minutes) |
| **Debrief OS** | Forensic memory | Reflective, honest | Past |
| **Portfolio OS** | Institutional awareness | Strategic, anticipatory | Season/quarter |

All four layers share the same operational truth engine.

## Planner OS cognition framework

The planner's dominant emotional state during orchestration is NOT emergency. It is:

1. **Anticipation** — what needs to happen next
2. **Readiness** — am I prepared for Saturday
3. **Coordination** — is everyone aligned
4. **Awareness** — what am I forgetting
5. **Confidence** — I have visibility into everything

## Orchestration hierarchy (what surfaces first)

1. Unresolved dependencies (blocking readiness)
2. Timing pressure (approaching deadlines)
3. Vendor acknowledgment states (who confirmed, who hasn't)
4. Client approval gates (decisions needed)
5. Staffing readiness (team availability)
6. Institutional patterns (learned from prior events)

## Runtime vs Planner philosophy split

- **Planner OS breathes more.** Less suppression, less compression, less emergency tension. More orchestration, more readiness, more anticipation, more confidence.
- **Planner OS shows names, not codes.** Vendor names, not status codes. Full descriptions, not compressed labels.
- **Planner OS surfaces dependencies, not emergencies.** Unresolved floral confirmation, not FLORAL EMERGENCY.
- **Planner OS includes whispers.** Institutional memory surfaces quietly in the right rail. Vendor reliability patterns. Venue quirks.
- **Planner OS shows the portfolio.** Multi-event coordination, timing compression, staffing load across events.
- **Runtime OS takes over when severity escalates.** The transition is a cognitive mode shift, not a page navigation.

## Drift risks identified

1. **Dashboard drift** — Portfolio overview could easily become KPI tiles. The surface shows unresolved items sorted by urgency, NOT metrics.
2. **CRM drift** — Client section could become pipeline tracking. It shows communication cadence and approval gates, NOT deal stages.
3. **Analytics drift** — Vendor memory could become scoring charts. It shows factual recall (whispers), NOT health percentages.
4. **Project management drift** — Timeline could become kanban. It shows day-of sequencing, NOT task boards.
5. **Startup SaaS drift** — The calm breathing room could invite pastel gradients and rounded corners everywhere. Studio Matte restraint must hold.

## What was intentionally refused

- Giant KPI tiles
- Analytics theater / trend charts
- AI insight cards / predictive scoring
- Productivity metrics / health percentages
- Kanban board thinking
- CRM pipeline stages
- Heatmaps / burndown charts / variance analysis
- Consumer productivity app aesthetics
- Pastel SaaS palette
- Cyberpunk operational fantasy
- Graphs of any kind

## What is NOT validated

- No real planner has reviewed these surfaces. All content is synthetic.
- The orchestration hierarchy is a hypothesis until field sessions confirm it.
- The Planner/Runtime transition has not been experienced by an operator.
- The whisper system content is fabricated for demonstration.
- None of these surfaces exist in code yet. They are Figma explorations.

## Honest read

This sprint establishes the cognitive system architecture that was missing. The project over-indexed on runtime containment for 6+ sprints. Planner OS corrects this by creating the orchestration layer that planners actually spend 95% of their time in. The four-layer model (Planner/Runtime/Debrief/Portfolio) gives each cognitive mode its own optimization target, preventing the design confusion that happens when you build crisis software for people who mostly orchestrate. The surfaces feel like institutional preparation, not dispatch software. Whether they hold up under real planner scrutiny is unknown.
